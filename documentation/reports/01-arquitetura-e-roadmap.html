1

Diagnóstico da arquitetura atual

O Estudix é hoje um app Expo/React Native (SDK 57, React 19, React Navigation 7) 100% client-side: sem backend, sem API, sem banco de dados relacional e sem autenticação além de um nome de exibição local. Toda a persistência é um único blob JSON em AsyncStorage (chave @Estudix:state), lido e escrito por um único contexto React.

Estrutura de pastas

src/
├─ components/   AppHeader, ConfirmModal, FAB, GradeChart, Toast
├─ context/      EstudixContext.js — estado global + ~30 mutações + persistência + haptics + toast, tudo num arquivo (781 linhas)
├─ lib/          notifications.js — notificações locais (expo-notifications)
├─ navigation/   RootNavigator (stack) + BottomTabNavigator (4 abas)
├─ screens/      9 telas, uma por rota, sem sub-componentização por domínio
└─ theme/        tokens de design (cores, tipografia, espaçamento, sombra)

Não existe camada services/, models/ ou data/ — regra de negócio, acesso a dado e efeito colateral (haptics, toast, notificação) vivem juntos dentro das funções do contexto, chamadas diretamente pelas telas via useEstudix().

Gerenciamento de estado, navegação e persistência

Estado: Context API pura (sem Redux/Zustand), um único Provider na raiz — qualquer dispatchUpdate re-renderiza toda a árvore que consome o contexto. Navegação: native-stack na raiz (Home/Materias/Foco/Anotações em bottom-tabs, e Calendário/Configurações/Matéria/Menu como rotas empilhadas). Persistência: debounce de 1s grava o estado inteiro a cada mudança; sem versionamento de esquema — compatibilidade retroativa é feita por checagens === undefined ad hoc dentro do loadStore.

Pontos fortes

  • Design tokens centralizados em src/theme/index.js — consistência visual real entre telas.
  • Persistência com debounce evita I/O excessivo a cada tick do timer.
  • Backup/restauração via JSON já funcional (exportData/importData).
  • Repetição espaçada (SM-2 simplificado) já implementada nos flashcards.
  • Conquistas e sequência de dias (currentStreak) já existem em focusSessions.
  • Notificações locais já ligadas ao Pomodoro e a eventos do calendário.

Pontos fracos

  • Contexto único ("God Context") sem separação por domínio.
  • Sem normalização — arrays achatados filtrados por materiaId em cada tela.
  • Sem schemaVersion/migração real do estado persistido.
  • IDs inconsistentes — só matérias têm prefixo (mat-).
  • Regra de negócio, I/O e efeito colateral misturados na mesma função.
  • Nenhum teste automatizado no projeto.
Risco principal

As nove funcionalidades pedidas introduzem pelo menos seis domínios de dado novos (perfil, catálogo, recomendações, plano de estudos, dashboard, IA). Empilhar tudo isso no mesmo EstudixContext.js e no mesmo blob de AsyncStorage tornaria o arquivo insustentável e infla cada backup do usuário com conteúdo que é igual para todo mundo.

Modelo de dados recomendado

Separação estrutural entre catálogo (estático, igual para todos, no bundle do app) e dados do usuário (mutáveis, no AsyncStorage). É a base de tudo o resto — sem ela, o plano de estudos e o dashboard não têm onde se apoiar sem inflar o backup pessoal do usuário.

Catálogo (somente leitura, bundle do app)

EducationLevelcatálogo

id, label, order — fundamental1, fundamental2, medio, tecnico, faculdade, pos, concurso, enem, vestibular, outro.

DisciplineCatalogEntrycatálogo

id (slug), name, aliases[], levels[], difficulty (1–5), description, importance, topics[], tips[], commonMistakes[], prerequisites[], recommendedOrder, avgWeeklyHours, techniques[] (cada uma tagueada por método — leitura/exercícios/resumo/vídeo), keywords[].

Dados do usuário (mutáveis, AsyncStorage)

UserProfilenovo

educationLevel (ref), goal ('enem'|'vestibular'|'concurso'|'faculdade'|'reforco'|'outro'), weeklyAvailability { totalMinutes, preferredDays[], preferredPeriod }, difficulties[] (tags: concentração, tempo, ansiedade, base fraca…), studyMethod ('leitura'|'exercicios'|'resumo'|'video'|'misto').

Materiaexistente + 1 campo

Estrutura atual mantida (id, name, icon, color) + campo opcional catalogSlug, nunca obrigatório.

StudyPlan → ScheduleBlocknovo

StudyPlan: id, generatedAt, snapshot do perfil usado na geração. ScheduleBlock: id, planId, materiaId, dayOfWeek/date, durationMin, priorityScore, source: 'auto-plan'. Blocos são materializados como calendar.events com sourcePlanId, para que regenerar o plano nunca toque em eventos criados manualmente pelo usuário.

Recommendation / InsightCardcalculado, não persistido

Derivações em runtime de materias + catálogo (recomendações) e de notas/flashcards/timer (insights do dashboard). Persistir seria dado duplicado e desatualizável.

Por que o catálogo fica fora do AsyncStorage

É idêntico para todos os usuários, vai crescer de dezenas para centenas de disciplinas, e atualizar seu conteúdo não deveria depender de migração de dados pessoais. A relação Materia → DisciplineCatalogEntry é muitos-para-um opcional por slug; prerequisites forma um grafo dirigido dentro do próprio catálogo, reaproveitado tanto pelas recomendações quanto pelo gerador de plano de estudos.

4

Estrutura de serviços

Uma camada nova, src/services/, com funções puras (sem Haptics, Toast, AsyncStorage ou navegação dentro delas) chamadas por hooks finos que combinam o resultado com o estado do contexto. As telas nunca chamam um serviço diretamente.

catalogService

Acesso somente-leitura ao catálogo: busca por slug, matching de nome/aliases, listagem por nível.

Entrada: nome digitado ou slug
Saída: DisciplineCatalogEntry | null
recommendationEngine

Cruza matérias do usuário com o catálogo (pré-requisitos, desempenho, revisões pendentes) e gera recomendações.

Entrada: materias, notas, catálogo
Saída: Recommendation[]
insightsEngine

Deriva os cards do dashboard — matéria crítica, hora de revisar, sequência em risco.

Entrada: estado do usuário
Saída: InsightCard[]
studyPlanGenerator

Heurística gulosa e determinística de alocação de tempo por prioridade — não é um solver de otimização exata. Reexecutável a qualquer momento (idempotente): mesma entrada, mesmo plano.

Entrada: profile, materias, catálogo, provas no calendário
Saída: ScheduleBlock[]
pomodoroStatsService

Agregações sobre focusSessions — hoje, semana, produtividade, sequência. Extraído do contexto para ser testável isoladamente.

Entrada: focusSessions[]
Saída: métricas agregadas
ai/AIProvider

Ponto único de extensão para IA futura: suggestPlan(), explainRecommendation(), answerQuestion(). Implementação inicial (LocalRulesProvider) apenas chama os serviços determinísticos acima.

Entrada: mesma dos serviços que envolve
Saída: mesma forma, venha de regra local ou de IA

Telas → hooks finos (useRecommendations, useInsights, useStudyPlan) → serviços puros → catálogo (bundle) + estado do usuário (AsyncStorage). Haptics, Toast e notificações continuam de responsabilidade das telas/hooks, nunca dos serviços.

Fluxo do usuário

Do primeiro contato ao uso diário, incluindo onde o plano de estudos e o dashboard entram na rotina.

  1. Onboarding — nome
    Passo já existente, sem alteração.
  2. Onboarding — perfil educacional (expandido)
    Nível, objetivo, disponibilidade semanal, dificuldades e método de estudo — como passos curtos de chips, não um formulário longo.
  3. Onboarding — sugestão de matérias
    Disciplinas comuns do nível/objetivo escolhido, marcáveis; cadastro livre continua disponível.
  4. Primeiro plano de estudos (opcional)
    Com o perfil recém-preenchido, o app oferece gerar um plano inicial; o usuário pode aceitar, ajustar ou pular.
  5. Dashboard (Home) — rotina diária
    Foco do dia, matéria crítica, revisões pendentes, progresso do plano — tudo priorizado, não uma lista extensa.
  6. Matéria Interna — aba Orientação
    Dificuldade, pré-requisitos, dicas e ordem recomendada, vindos do catálogo.
  7. Recomendações cruzadas
    Nota baixa em um pré-requisito gera sugestão tanto na Home quanto na matéria dependente.
  8. Foco — Pomodoro
    Introdução na primeira visita; sessões podem se vincular ao bloco do plano do dia, quando existir um.
  9. Regeneração do plano
    Mudança de disponibilidade ou objetivo permite gerar um novo plano sem apagar eventos manuais do calendário.

Melhorias de UX/UI

Ancoradas na linguagem visual que o Estudix já tem — papel quente, azul-marinho como primária, terracota como destaque, serifada nos títulos, chips em pílula — para que as novas telas pareçam parte do mesmo app.

Onboarding

Reaproveitar o padrão de dots de progresso (mais passos, mesmo componente). Nível, objetivo e método como grades de chips, no estilo do materiaChip da tela de Foco. Dificuldades como seleção múltipla de tags curtas.

Badge de dificuldade

Reaproveitar o padrão de mediaBadge (cores por faixa, já usado em notas) aplicado ao campo difficulty do catálogo.

Dashboard

Carrossel horizontal no padrão do bloco de conquistas (achievementsRow), no máximo um card de recomendação e uma dica por vez. Seções priorizadas: crítico primeiro, depois progresso, depois estatísticas — não uma lista plana.

Plano de estudos no calendário

Blocos gerados automaticamente aparecem na tira e no mês com a cor da matéria, mas visualmente distintos dos eventos manuais (ex.: borda tracejada ou um pequeno ícone "auto") — o usuário precisa reconhecer de relance o que ele criou e o que o sistema sugeriu.

Aba Orientação e introdução ao Pomodoro

Mesmo componente de abas já usado em MateriaInternaScreen.js; texto explicativo do Pomodoro reaproveitando o ConfirmModal existente.

8

Riscos técnicos

O que pode dar errado em cada frente, e como reduzir o risco sem reescrever o app.

Contexto monolítico crescendo
Alto impacto
Cada domínio novo somado ao EstudixContext.js aumenta re-renders e o custo de manutenção do arquivo.
Mitigação: dividir por domínio antes da Fase 3, não depois.
Lógica de datas já frágil
Alto impacto
O loadStore atual zera totalMinutesWeek comparando currDate.getDay() < lastDate.getDay() como proxy de "virou a semana" — isso quebra perto de virada de mês. O plano de estudos vai depender pesadamente de datas corretas para agendar blocos.
Mitigação: corrigir essa comparação antes de construir o gerador de cronograma sobre ela.
Conflito entre eventos manuais e blocos automáticos
Médio impacto
Regenerar o plano pode sobrescrever ou duplicar eventos que o usuário criou manualmente no calendário.
Mitigação: campo source/sourcePlanId nos eventos — regeneração só toca no que ela mesma criou.
Complexidade do gerador de cronograma
Médio impacto
Tentar uma alocação "ótima" (solver) é overengineering para o problema e imprevisível de depurar.
Mitigação: heurística gulosa simples e determinística, documentada, não um otimizador genérico.
Confiabilidade de notificações locais
Médio impacto
Permissão negada ou app encerrado pelo SO faz o lembrete de um bloco do plano simplesmente não disparar, sem fallback dentro do app.
Mitigação: sempre mostrar o bloco do dia no dashboard, independente da notificação ter disparado.
Catálogo em escala
Baixo impacto hoje
Matching de nome ingênuo fica impreciso conforme o catálogo cresce de dezenas para centenas de disciplinas.
Mitigação: normalizar + índice por aliases/keywords desde a Fase 1, mesmo com poucas entradas.
Integração futura de IA quebra a promessa de app offline
Risco futuro
Custo, latência e dependência de rede não existem hoje no Estudix; uma chamada de IA mal isolada tornaria features hoje instantâneas em algo que espera resposta de rede.
Mitigação: AIProvider sempre com fallback determinístico local, nunca bloqueante.
9

Recomendações finais

Prioridade para soluções simples, desacopladas e de fácil manutenção — sem overengineering.

  • Quebrar o contexto monolítico em contextos por domínio (perfil, matérias, catálogo, plano, pomodoro) antes de somar mais funcionalidades a ele.
  • Manter o catálogo fora do AsyncStorage — asset estático no bundle, nunca no backup exportado do usuário.
  • Introduzir a camada de serviços puros descrita na seção 4 — testável isoladamente, sem efeitos colaterais, reaproveitável.
  • Gerador de plano como heurística simples, não solver — determinístico, idempotente, fácil de explicar ao usuário ("por que este horário?").
  • Versionar o esquema do estado (schemaVersion + migrate(state)) em vez dos remendos === undefined atuais — vira crítico assim que StudyPlan e UserProfile entrarem no blob.
  • Preparar a interface de IA desde a Fase 1, com implementação puramente determinística — evita retrabalho na Fase 5.
  • Manter a interface enxuta: a inteligência aparece como orientação pontual dentro do que já existe, nunca como um módulo novo e pesado.
Relatório de análise arquitetural — nenhuma alteração de código foi feita. Próximo passo natural: validar as Fases 1–2 do roadmap antes de qualquer implementação.
