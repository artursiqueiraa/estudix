1

Melhor arquitetura para essa funcionalidade

O primeiro ponto a deixar claro: o gargalo real desta funcionalidade não é engenharia — é dado. A BNCC nacional é um documento público, mas não vem pronta em JSON; currículos estaduais têm qualidade e disponibilidade digital muito desiguais entre os 27 estados; "concurso" e "vestibular" nem têm um programa único (cada edital define o seu). A arquitetura certa é a que funciona bem com cobertura parcial desde o dia um e melhora conforme mais currículos forem curados — nunca uma que finja ter os 27 estados prontos de largada.

A cadeia de resolução (fallback), não um "currículo único"

Em vez de exigir um currículo estadual completo para cada combinação de UF/rede/série, o sistema resolve em camadas, da mais específica para a mais genérica, e usa a primeira que existir:

Mais específico
Currículo do exame
Editais de concurso/vestibular específicos, quando curados
→
 
Currículo estadual
Ex.: Currículo Paulista, quando disponível para aquela UF+série
→
Sempre disponível
BNCC nacional
Baseline garantida para toda modalidade/série do país
→
Último recurso
Disciplina genérica
Catálogo de disciplinas já previsto no roadmap técnico, sem amarração curricular

Essa cadeia é uma função pura — resolveCurriculo(perfil) — que nunca falha: na pior hipótese, cai no catálogo genérico já planejado. Isso também resolve directamente o "quando existirem" que está no seu próprio pedido — não é uma frase de segurança, é literalmente a arquitetura.

Três camadas, três taxas de mudança diferentes

Currículo (estático, bundle)

  • Muda uma vez por ano (ciclo letivo), não por usuário
  • Igual para todo mundo com o mesmo perfil
  • Nunca entra no AsyncStorage nem no backup do usuário
  • Particionado por UF+modalidade, carregado sob demanda (só o do perfil do usuário)

Provisionamento (acontece uma vez)

  • Resolve o currículo → cria as matérias do usuário
  • É o único momento em que "currículo" e "dado do usuário" se tocam
  • A partir daqui, o que foi criado é do usuário — igual a uma matéria criada manualmente

Dado do usuário (AsyncStorage, já existente)

  • materias, checklistCategories, flashcards, anotacoes — as mesmas coleções que já existem hoje, sem novas tabelas para "conteúdo pré-carregado"
  • Editável, apagável, substituível — porque é uma linha comum dessas coleções, não uma cópia viva do currículo
Por que isso é a decisão certa aqui

O pedido é explícito: "o sistema nunca deve substituir automaticamente conteúdos criados pelo usuário". A única forma de garantir isso com certeza arquitetural — não como uma regra de negócio frágil — é o conteúdo pré-carregado nascer como dado comum do usuário (materializado) em vez de viver como uma referência ao currículo que é resolvida toda vez que a tela abre. Depois de materializado, editar um flashcard "sugerido" é EXATAMENTE o mesmo código que já edita um flashcard criado manualmente — nenhuma lógica nova de "isso é especial, cuidado ao editar".

3

Estrutura do catálogo curricular

A pergunta que decide a arquitetura aqui não é "que campos tem cada disciplina" — é "como isso não vira um app de 2GB". Fazendo as contas: 27 UFs × ~4 modalidades relevantes × ~3 séries × ~10 disciplinas × conteúdo rico por tópico é, com folga, grande demais para embutir inteiro em todo APK/IPA.

Particionamento por UF + modalidade, carregado sob demanda

Cada CurriculumDocument vive em seu próprio arquivo JSON (src/data/curriculum/sp/medio-3ano.json, src/data/curriculum/bncc/medio-3ano.json...). O app nunca carrega o Brasil inteiro — resolve o perfil do usuário e faz um import dinâmico só do(s) documento(s) relevantes. Isso mantém o bundle inicial do app pequeno independente de quantos estados forem sendo cobertos com o tempo.

BNCC como piso, currículo estadual como cobertura

CamadaCobertura hojeFonte
BNCC nacional100% do paísDocumento público estruturado (habilidades codificadas), digitalização única e reaproveitável para todas as UFs
Currículos estaduaisDesigual entre estadosAlguns estados publicam currículo próprio digitalizável; outros não têm nada além da BNCC — nesses casos o fallback já cobre sem quebrar nada
Concurso / VestibularCaso a casoSem programa nacional único — cada edital define o seu; tratar com granularidade de "grandes áreas", não fingir profundidade BNCC que não existe para esse objetivo

De onde vem o trabalho de curadoria

Isso não é um problema que se resolve só com código — é um projeto de conteúdo. A digitalização estruturada de currículos (extrair de PDFs oficiais para o formato ContentTopic) é candidata natural a um pipeline semi-automatizado com revisão humana, usando a mesma camada de abstração de IA já prevista no roadmap técnico (AIProvider) — não para gerar conteúdo pedagógico sem revisão, mas para acelerar a extração/estruturação inicial que depois passa por curadoria antes de entrar no bundle.

4

Fluxo completo — do cadastro à disciplina pronta

  1. Perfil educacional expandido
    Além do que já existe (nível, objetivo), o onboarding pergunta UF, rede de ensino, série, turno e — se aplicável — semestre/bimestre atual e data da prova.
  2. Resolução do currículo
    resolveCurriculo(perfil) percorre a cadeia de fallback (seção 1) e retorna um ou mais CurriculumDocument — sempre retorna algo, nunca falha.
  3. Provisionamento das matérias
    Uma única ação nova de contexto (ex. provisionMateriasFromCurriculum) cria todas as matérias do currículo resolvido num só dispatchUpdate — reaproveitando a mesma lógica de cor/ícone cíclico que saveMateria já usa, não uma reimplementação.
  4. Evento de prova no calendário (quando informado)
    saveEvent(..., type: 'prova') — a mesma função e o mesmo tipo de evento que já existem em CalendarioScreen.js.
  5. "Começar a Estudar" — materialização sob demanda
    No provisionamento, cada matéria carrega só a referência ao currículo (leve). O conteúdo pesado (checklist, flashcards, resumo) só é criado de fato na primeira vez que o usuário abre aquela matéria — evita inflar o backup do usuário com conteúdo de 12 matérias que ele talvez nunca abra.
  6. Quiz diagnóstico (opcional)
    Poucas perguntas de sim/não; o resultado vira materia.diagnostic.weakTopicIds, consumido pelo motor de recomendações e pelos insights do dashboard já previstos no roadmap técnico.
  7. Personalização contínua
    A partir daqui, tudo é dado comum do usuário. Editar, apagar ou substituir um flashcard "sugerido" é o mesmo fluxo — e o mesmo código — de sempre.
5

Estratégia para manter os currículos atualizados

O princípio que já rege o resto do app se aplica aqui sem exceção: atualizar o catálogo nunca pode alterar silenciosamente dado que o usuário já tem.

  • Versionamento por vigência. Cada CurriculumDocument tem um ano de vigência. Um usuário que já provisionou suas matérias em 2026 continua com as matérias de 2026 mesmo depois do currículo 2027 chegar — a atualização só afeta novos provisionamentos.
  • MVP: bundle estático, atualizado a cada release do app. Suficiente para o ciclo anual de currículo — não é algo que muda no meio do ano letivo na prática.
  • Evolução natural: fetch remoto com cache local. Quando o catálogo crescer o bastante para não caber mais em atualizações de app store (ou precisar de correção fora do ciclo de release), o mesmo catalogService já planejado no roadmap técnico ganha uma fonte remota com cache — sem mudar quem o consome. Não construir isso agora seria overengineering; a estrutura já fica pronta para essa migração sem retrabalho.
  • Correções pontuais não exigem migração de dado do usuário — só afetam o bundle de currículo, que é somente leitura.
6

Impacto na experiência do usuário

  • Ativação imediata. O tempo entre "abrir o app pela primeira vez" e "ver uma disciplina com conteúdo de verdade" cai de "cadastrar tudo manualmente" para minutos — ataca de frente o principal ponto de atrito já identificado no relatório de produto (tela vazia no primeiro contato).
  • Onboarding mais longo, mas opcional em cada campo. UF/rede/série/turno somam perguntas ao cadastro já existente (que passou de 2 para 3 passos na Feature 1). Cada campo novo deve permanecer pulável — o app já funciona sem eles, só com resolução menos específica na cadeia de fallback.
  • Risco de sobrecarga na primeira abertura de cada matéria. Doze matérias "prontas" de uma vez podem parecer spam se aparecerem todas cheias ao mesmo tempo — mitigado pela materialização sob demanda (seção 4): cada matéria só ganha conteúdo pesado quando o próprio usuário a abre.
  • Diagnóstico precisa de tom cuidadoso. Sinalizar "nível baixo" de cara pode desmotivar — o enquadramento importa tanto quanto os dados (ecoa a recomendação do relatório de produto sobre gamificação ética: mastery, não julgamento).
7

Riscos técnicos

Tamanho do bundle
Alto impacto
Currículo rico multiplicado por 27 estados facilmente ultrapassa um tamanho razoável de app.
Mitigação: particionamento por UF+modalidade com import dinâmico (seção 3) — nunca carregar o Brasil inteiro de uma vez.
Cobertura desigual entre estados
Alto impacto
MVP provavelmente cobre bem 2–3 estados e cai para "só BNCC genérico" nos demais — expectativa desigual entre usuários de estados diferentes.
Mitigação: a cadeia de fallback já garante que "genérico" nunca é "quebrado" — e a UI deve deixar claro qual camada foi usada, sem prometer o que não tem.
Rede pública vs. privada na prática
Médio impacto
Escolas privadas frequentemente seguem material apostilado próprio, não o currículo estadual oficial — "rede de ensino" no perfil pode sugerir uma precisão que o dado não tem.
Mitigação: tratar "rede" como ajuste de tom/ênfase das dicas, não como fonte de conteúdo apostilado específico — isso é um mercado à parte, fora de escopo.
Auto-criação em massa sentida como spam
Médio impacto
12 matérias com dezenas de itens cada aparecendo de uma vez pode sobrecarregar visualmente.
Mitigação: materialização sob demanda (seção 4), já parte do desenho, não um ajuste posterior.
Internacionalização
Risco futuro
A estrutura de CurriculumDocument (escopo nacional/estadual/exame) já comportaria trocar o bundle por outro país, mas a UI do app inteiro é hoje hardcoded em português (por instrução explícita do projeto).
Mitigação: não investir em i18n de UI especulativamente agora — o modelo de dado já não impede isso depois, e é só isso que precisa ser verdade hoje.
8

Diferencial competitivo

Nenhum dos concorrentes já mapeados no relatório de produto anterior — Forest, Anki, StudySmarter, Todoist/TickTick, Notion, Habitica, Google Agenda — tem currículo brasileiro oficial embutido. É o tipo de diferencial difícil de copiar rápido, porque a barreira é curadoria de dado, não código.

  • É o "momento de tutor pessoal" levado ao extremo: abrir o app pela primeira vez e a matéria já vir com conteúdo relevante é a versão mais forte possível do "esse app entende como eu estudo" identificado no relatório de produto.
  • Monetização já desenhada, agora com um gancho natural: BNCC nacional gratuito para todos; cobertura estadual completa e atualização prioritária como parte do tier Premium já recomendado — sem nunca cobrar pela ferramenta central de estudar, consistente com a estratégia definida anteriormente.
  • Efeito de rede de conteúdo: conforme mais currículos estaduais forem curados, o produto melhora para todo mundo daquele estado de uma vez — diferente de features que só beneficiam quem as usa individualmente.
